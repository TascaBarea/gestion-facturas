## REGLAS CRГҚTICAS (errores documentados del proyecto)

### Excel y openpyxl
- **[CRГҚTICO] Excel abierto al escribir** вҶ’ fallo silencioso o corrupciГіn de archivo
  вҶ’ REGLA: Avisar SIEMPRE al usuario antes de cualquier escritura Excel. Sin excepciГіn.
- **Sobreescritura sin leer primero** вҶ’ pГ©rdida de datos existentes
  вҶ’ REGLA: save_to_excel() SIEMPRE lee hoja existente antes de escribir. Dedup por unique_col.
- **Backup omitido** вҶ’ sin posibilidad de recuperaciГіn si algo falla
  вҶ’ REGLA: Backup automГЎtico en datos/backups/ antes de la primera escritura de cada sesiГіn

### Scripts batch (.bat)
- **ERRORLEVEL dentro de bloques if ()** вҶ’ no se actualiza en Windows
  вҶ’ REGLA: Usar goto + labels para manejo de errores. Nunca if ERRORLEVEL anidado.
- **PYTHONPATH no configurado en gmail_auto.bat** вҶ’ `ModuleNotFoundError: nucleo` tras reorganizaciГіn de imports
  вҶ’ REGLA: Si un script usa mГіdulos del proyecto (nucleo/, config/), el bat debe hacer `set "PYTHONPATH=%PROJECT_ROOT%"` antes de ejecutar Python. Verificar .bat tras cualquier cambio de imports.

### Gmail API
- **Token OAuth2 caducado** вҶ’ error silencioso o crash en gmail.py
  вҶ’ REGLA: Si gmail.py falla con error de autenticaciГіn вҶ’ ejecutar gmail/renovar_token_business.py
- **MIME type con barra invertida** вҶ’ text\html en vez de text/html
  вҶ’ REGLA: Al parsear payloads Gmail API, verificar que el MIME type usa barra normal.
- **Cambio de scope OAuth no se aplica con solo editar SCOPES** вҶ’ la librerГӯa usa los scopes persistidos dentro de `token.json`, no los del cГіdigo. SГӯntoma: 403 "insufficient authentication scopes" aunque el cГіdigo declare el scope nuevo.
  вҶ’ REGLA: Al cambiar scope, borrar/renombrar `token.json` (`mv token.json token.json.bak.YYYYMMDD`), ejecutar `python gmail/renovar_token_business.py` (navegador OAuth), copiar el nuevo `token.json` al VPS con `scp`.
- **SerializaciГіn de credenciales filtra scopes вҶ’ se pierden al refresh** вҶ’ Pasar `scopes=[...]` a `Credentials.from_authorized_user_file(token, scopes=X)` hace que, al refrescar y volcar con `creds.to_json()`, se persista SOLO el subconjunto `X`, destruyendo los scopes realmente autorizados. Caso real (21/04/2026): el scope `drive` desapareciГі de `token.json` entre el 18/04 y el 20/04 porque cada run de gmail.py pasaba solo `[gmail.readonly, gmail.modify]` y reescribГӯa el token.
  вҶ’ REGLA: En `gmail/auth_manager.get_credentials()` cargar sin parГЎmetro `scopes`; dejar que el token dicte quГ© estГЎ autorizado. Los callers (`get_gmail_service`, `get_drive_service`) tampoco deben pasar scopes.
  вҶ’ **REGLA ALTERNATIVA** (cuando NO se puede evitar pasar scopes вҖ” caso `gmail/gmail.py:706` que `auth_manager` no controla): listar TODOS los scopes que el token tiene autorizados, no solo los del mГіdulo concreto. La lista pasada a `from_authorized_user_file()` es lo que `creds.to_json()` persistirГЎ tras refresh вҖ” si pones un subconjunto, persistes un subconjunto.
  вҶ’ REVISITADO (04/05/2026): el mismo bug reapareciГі porque solo se parcheГі `auth_manager.py` el 21/04, pero `gmail.py:706` seguГӯa pasando `CONFIG.GMAIL_SCOPES = [gmail.readonly, gmail.modify]` (sin drive). Cada ejecuciГіn de gmail.py refrescaba y borraba `drive` del token. SГӯntoma: scope desaparece entre ejecuciones sin que nadie lo haya tocado. DetecciГіn: tras un fix de OAuth, `grep -rn "from_authorized_user_file" gmail/ scripts/ streamlit_app/` para asegurar que TODOS los call sites o pasan `None` o pasan la lista completa de scopes del proyecto.
  вҶ’ RESUELTO (05/05/2026): parche en `gmail/gmail.py:191-194` aГұadiendo `drive` a `GMAIL_SCOPES`. Token regenerado con 4 scopes (gmail.readonly + gmail.modify + business.manage + drive). Drive API verificada (list carpeta canГіnica OK). Test robustez: `creds.to_json()` preserva los 4 scopes tras load. Deuda tГ©cnica: refactor de `gmail.py:conectar()` para delegar en `auth_manager.get_gmail_service()` y eliminar la duplicaciГіn permanentemente. Mientras existan dos call sites paralelos de `from_authorized_user_file`, **cualquier scope nuevo del proyecto debe aГұadirse a TODAS las listas** o el bug volverГЎ.

### Formato de datos
- **Coma decimal en datos 2024** вҶ’ "3,51" en vez de "3.51"
  вҶ’ REGLA: Al leer Excel histГіrico 2024, convertir comas a puntos antes de operar.
  вҶ’ REGLA: Datos 2025+ usan punto. No aplicar conversiГіn si el archivo es de 2025+.
- **Formato moneda en Excel/email** вҶ’ SIEMPRE usar formato europeo: X,xx вӮ¬ (coma decimal, sГӯmbolo al final)
  вҶ’ REGLA: Al escribir importes en Excel o HTML, usar `f"{v:.2f}".replace(".", ",") + " вӮ¬"`
  вҶ’ REGLA: Fechas en columnas Excel de ventas: formato DD-MM-YY (no YYYY-MM-DD)

### Extractores PDF
- **Extractor nuevo sin manejo de excepciones** вҶ’ factura sin procesar si falla
  вҶ’ REGLA: Todo extractor debe devolver datos parciales ante fallo, nunca lanzar excepciГіn.
- **OCR como mГ©todo primario** вҶ’ solo para PDFs imagen pura confirmados
  вҶ’ REGLA: Usar OCR primario solo en extractores confirmados: JIMELUZ, CASA DEL DUQUE, LA LLILDIRIA.
  вҶ’ Para el resto: pdfplumber o pypdf con fallback_ocr=True.
- **`extraer_numero_factura` eliminado en Parseo v5.18** вҶ’ gmail.py debe usar `extraer_referencia`
  вҶ’ REGLA: Al crear extractores nuevos, solo implementar `extraer_referencia` (nunca `extraer_numero_factura`).
- **MAESTRO con `ARCHIVO_EXTRACTOR=X.py` pero archivo inexistente** вҶ’ gmail.py loguea `Extractor no encontrado` y cae al fallback genГ©rico, dejando filas Excel zombi (TOTAL/REF vacГӯos). Caso real (04/05/2026): `ACEITES DE ESPECIALIDAD JALEO` con `ARCHIVO_EXTRACTOR=jaleo.py` configurado en MAESTRO desde hacГӯa meses, pero `Parseo/extractores/jaleo.py` nunca se creГі вҶ’ 2 zombis acumuladas.
  вҶ’ REGLA: Al dar de alta un proveedor en MAESTRO con `TIENE_EXTRACTOR=SI` y `ARCHIVO_EXTRACTOR=X.py`, crear el archivo el mismo dГӯa. Si se pospone, marcar `TIENE_EXTRACTOR=NO` hasta que estГ© listo, asГӯ gmail.py usa fallback genГ©rico explГӯcito (no roto).
  вҶ’ REGLA: Para detectar futuros casos, ejecutar periГіdicamente: para cada fila MAESTRO con `TIENE_EXTRACTOR=SI`, verificar `Path(extractores_dir / archivo_extractor).exists()`. Discrepancia вҶ’ alerta.
- **Mismo proveedor con 2 formatos PDF distintos** вҶ’ un solo regex no cubre ambos casos. Caso real (04/05/2026): JALEO tiene formato "factura moderna" (cabecera one-line `Factura N\d+ Fecha dd/mm/yyyy ... Total XвӮ¬`) y formato "presupuesto" (bloques separados con `NГҡMERO DE ORDEN #\d+`, `FECHA DE ORDEN dd-mm-yyyy`, `TOTAL : X вӮ¬`).
  вҶ’ REGLA: Cuando un proveedor tiene >1 formato, escribir CADA mГ©todo (`extraer_total/fecha/referencia`) con regex jerГЎrquicos: primero el patrГіn mГЎs especГӯfico, luego el genГ©rico. Validar smoke test con un PDF de cada formato. Documentar los formatos en el docstring del mГіdulo.
  вҶ’ REGLA: Cuidar colisiones con campos parecidos. Caso JALEO: `extraer_total` debe matchear `TOTAL : 523,50 вӮ¬` pero NO `TOTAL PARCIAL : 698,00 вӮ¬` вҶ’ usar `^TOTAL\s*:` con `re.MULTILINE`. Y `extraer_referencia` debe priorizar `Factura N\d+` sobre `#\d+` porque PDF1 contiene `Observaciones #1859` que NO es la REF real.
- **PDFs con jerarquГӯa visual cliente/proveedor invertida** вҶ’ la heurГӯstica genГ©rica de gmail.py registra el CLIENTE como proveedor. Caso real (04/05/2026): factura COMPROVINO con cabecera maquetada poniendo "COMESTIBLES BAREA" + "Elena de Miguel" + email cliente en posiciГіn visualmente prominente, mientras "COMPROVINO SL / BODEGABIERTA" aparecГӯa a la izquierda con jerarquГӯa menor. El CIF B85501989 es del proveedor pero acabГі asociado al "proveedor nuevo" COMESTIBLES BAREA en el log.
  вҶ’ REGLA: cuando el log diga `Proveedor NUEVO ... CIF X` y X coincida con un CIF cliente conocido (B87760575 para Tasca Barea SLL en este negocio), hay alta probabilidad de bug de confusiГіn cliente/proveedor. Investigar el PDF original. SoluciГіn: extractor dedicado con `cif` estГЎtico (no extraer del PDF вҖ” los 2 CIFs suelen estar en la misma lГӯnea y separarlos sin contexto es frГЎgil) + alta MAESTRO con alias mГәltiples (razГіn social + nombre comercial; ambos llegarГЎn como remitente en distintos emails).
  вҶ’ REGLA: smoke test del extractor debe incluir aserto defensivo: `assert extractor.cif == "<CIF_PROVEEDOR>"` y `assert extractor.cif != "<CIF_CLIENTE>"`. Si en el futuro alguien cambia el CIF estГЎtico por accidente o alguien aГұade `extraer_cif` que captura el CIF cliente, el smoke test lo pilla.

### Tests obsoletos tras refactor de la API testeada
- **Tests CI fallan en collection con `AttributeError: module 'X' has no attribute 'Y'`** вҶ’ la API que el test importa fue renombrada/eliminada en algГәn refactor anterior y los tests no se actualizaron. SГӯntoma tГӯpico: tests llevan tiempo rotos pero nadie los arregla porque estГЎn mezclados con deuda CI mГЎs amplia.
  вҶ’ REGLA: cuando un test rompe en collection con `AttributeError`/`ImportError` sobre sГӯmbolos del cГіdigo de producciГіn, el primer paso es `git log -S "<simbolo>" -- <archivo>` para ver dГіnde se eliminГі. Suele apuntar a un merge o refactor concreto. Ese commit decide si la feature fue descartada (вҶ’ borrar tests) o si fue regresiГіn (вҶ’ restaurar).
  вҶ’ REGLA: distinguir B (constante perdida) vs A (tests obsoletos) por uso en producciГіn: `grep -rn "<simbolo>" --include="*.py" src/` (excluyendo tests). Si solo aparece en tests вҶ’ tests obsoletos, borrar. Si aparece tambiГ©n en cГіdigo de producciГіn вҶ’ regresiГіn, restaurar.
  вҶ’ REGLA: **al hacer merge con conflictos donde se prioriza un lado completamente** (caso 18/04/2026 cec9306 "prioridad cГіdigo PC"), revisar la suite de tests antes de empujar вҖ” los tests del lado descartado se quedarГЎn colgados. Es deuda que se acumula silenciosa hasta el siguiente push que dispara CI.
- **Errores en collection enmascaran otros errores**: cuando collection muere al primer fallo, los tests posteriores ni se intentan. Caso real (05/05/2026): tras borrar los 2 tests obsoletos por `TOTAL_MIN_SOSPECHOSO`, en CI aparecieron 4 ImportError NUEVOS de `google`, `numpy`, `dotenv`, `googleapiclient` (deps que `pip install -e ".[dev]"` no cubre porque el workflow las omite del extra-install). En local tambiГ©n afloraron 22 fallos por falta de `pytest-asyncio` que estaban ocultos. Total: 26 problemas preexistentes que llevaban semanas tapados detrГЎs de 2 ImportError.
  вҶ’ REGLA: tras arreglar un fallo de collection, ejecutar la suite COMPLETA y leer la salida tanto en local como en CI. No dar por bueno "los tests pasan" si CI sigue rojo por motivos distintos. Documentar como backlog separado, no expandir scope del fix actual.
  вҶ’ REGLA: divergencia entre venv local y workflow CI suele ser fuente de sorpresas. Si el workflow tiene un `pip install` con extras manuales ademГЎs de `pip install -e ".[dev]"`, esos extras son seГұal de que el extra `dev` estГЎ incompleto y se van aГұadiendo deps a mano. Refactor recomendado: consolidar TODAS las deps necesarias para tests en `pyproject.toml [project.optional-dependencies] dev`.
  вҶ’ APLICADO (05/05/2026 noche-3): consolidaciГіn realizada. 11 deps movidas/aГұadidas a `[dev]` (google-api-python-client, google-auth, google-auth-oauthlib, numpy, openpyxl, pdfplumber, pydantic, python-dotenv, rapidfuzz, pandas, fastapi, python-multipart) + workflow reducido a `pip install -e ".[dev]"`. Suite local pasГі de 114 passed + 22 fail a 136 passed + 0 fail. **CI requiriГі 3 iteraciones** para verde porque cada push descubrГӯa una dep transitiva mГЎs que el venv local sГӯ tenГӯa: `1de22b5` (consolidaciГіn) вҶ’ `df71401` (+pandas+fastapi) вҶ’ `ab94d04` (+python-multipart) вҶ’ success 179 passed.
  вҶ’ REGLA: tras consolidar deps en `[dev]`, antes de hacer push hacer dry-run mental de "ВҝquГ© tengo en mi venv que NO estГ© en el repo?" вҖ” `pip freeze | diff <(pip-extract-deps pyproject.toml [dev])` puede ayudar. La trampa repetida: deps transitivas que no son obvias del cГіdigo (FastAPI necesita `python-multipart` en runtime para form data, pero no aparece en imports del test). Iterar reactivamente cada vez que CI revele una nueva es vГЎlido pero costoso (cada iteraciГіn = 1-2 min).

- **Cuando una auditorГӯa propone heurГӯsticas (regex, scoring, capas de prioridad), validarlas en frГӯo contra datos reales antes de aplicar el fix completo**. Caso real (06/05/2026 Bloque B): la auditorГӯa inicial proponГӯa 3 capas para `_nombre_aproximado`. Al analizar 2 PDFs reales del backlog (Torres Import + AquГӯ SantoГұa), descubrimos que se necesitaba una 4ВӘ capa (sufijo societario) y un bug ADICIONAL no contemplado (filtrado no-factura para catГЎlogos comerciales).

  вҶ’ REGLA: cuando se descubren вүҘ2 problemas relacionados durante validaciГіn, evaluar si agruparlos en una sesiГіn combinada (Bloque B + filtrado no-factura) o separarlas. La agrupaciГіn tiene sentido si el blast radius solapa (ambos tocaban `_procesar_pdf`).

### Tests con __init__ pesado en CI: usar __new__ para skipping
- **Tests que invocan clases con `__init__` que carga archivos del repo gitignored fallan en CI runner**. Caso real (06/05/2026): `_make_processor()` invocaba `GmailProcessor(modo_test=True)` que carga `MaestroProveedores(CONFIG.MAESTRO_PATH)`. MAESTRO estГЎ gitignored вҶ’ existe en local pero NO en CI runner вҶ’ `FileNotFoundError`.
  вҶ’ REGLA: si los mГ©todos testeados NO necesitan los atributos creados en `__init__`, usar `Clase.__new__(Clase)` para crear instancia sin disparar `__init__`. SoluciГіn mГӯnima sin mocks ni fixtures complejos.
  вҶ’ REGLA detecciГіn: tests pasan local, fallan CI con `FileNotFoundError` apuntando a archivo gitignored вҶ’ casi siempre es `__init__` que carga ese archivo. Verificar antes de ir a fixtures elaboradas.

### Lectura вү  escritura: paths divergentes en MAESTRO/datos compartidos
- **Cuando un consumidor (gmail.py) lee de una ruta y los scripts del repo escriben en otra, las altas se pierden silenciosamente**. Caso real (06/05/2026): gmail.py v1.21+ leГӯa `G:\Mi unidad\Barea - Datos Compartidos\Maestro\MAESTRO_PROVEEDORES.xlsx` (Drive Desktop) en Windows; los scripts de alta (`_alta_*.py`, manual de proveedor) escribГӯan en `c:\...\gestion-facturas\datos\MAESTRO_PROVEEDORES.xlsx`. Las altas COMPROVINO (05/05) y JALEO (sesiГіn anterior) llegaron a `datos/` pero NO a Drive. La prГіxima factura COMPROVINO o JALEO habrГӯa producido "Proveedor NUEVO" repitiendo el bug del 04/05.
  вҶ’ REGLA: cualquier dato compartido (MAESTRO, diccionarios, Гӯndices) tiene que tener UNA fuente de verdad y todos los productores y consumidores deben apuntar al mismo path. PatrГіn a vigilar: lectura desde Drive/Cloud + escritura desde scripts del repo en `datos/`.
  вҶ’ REGLA detecciГіn: cualquier alta de proveedor que no aparezca en la siguiente ejecuciГіn del consumidor вҶ’ bug de paths divergentes. Comprobar `grep -rn "MAESTRO_PROVEEDORES.xlsx" .` y verificar que TODOS los call sites resuelven al mismo archivo.
  вҶ’ REGLA soluciГіn: con 1 usuario, unificar en repo (`datos/`). Con multi-usuario, unificar en Cloud Y corregir scripts de alta para que escriban ahГӯ (no en repo). Migrar BвҶ’A o AвҶ’B es trivial (1 commit aislado) si se diseГұa bien la abstracciГіn `resolver_<recurso>_path`. Caso 06/05 v1.24: simplificaciГіn a 80 LOC menos + decisiГіn documentada en `outputs/fix_maestro_fuente_verdad_20260506.md`.

### Sync VPS вҖ” line endings CRLF vs LF
- **md5 de un archivo Python en PC (CRLF) difiere del md5 en VPS (LF)** вҖ” comportamiento normal de git con `core.autocrlf` en Windows. Caso real (06/05/2026): tras push de `gmail.py` v1.23, md5 PC `7798cc9e...` вү  md5 VPS `24efb225...`. NO indica fallo de sync.
  вҶ’ REGLA: para archivos de texto (`.py`, `.md`, `.yml`), validar sync VPS por contenido lГіgico (line count + grep de markers Гәnicos), no por md5. Para binarios (`.xlsx`, `.pdf`), md5 sГӯ debe coincidir porque git no los toca.

### Pre-checks de ejecuciГіn manual gmail.py
- **Lock files tambiГ©n en `outputs/`** вҖ” gmail.py escribe a `outputs/PAGOS_Gmail_<periodo>.xlsx` (copia local del repo, ademГЎs de Drive y Dropbox). Si Excel deja un lock zombi tras un crash en `outputs/~$PAGOS_Gmail_<periodo>.xlsx`, gmail.py aborta con `вқҢ ARCHIVO BLOQUEADO`. Caso real (05/05/2026 23:52): primer intento fallГі por `outputs/~$PAGOS_Gmail_2T26.xlsx` con mtime de 4h antes (Jaime ya habГӯa cerrado, lock zombi). Borrado con `rm` y relanzado OK.
  вҶ’ REGLA: pre-checks de lock files DEBEN incluir `outputs/` ademГЎs de `G:\Drive` y `Dropbox`. Las 3 ubicaciones tienen impacto distinto: locks en Drive/Dropbox bloquean SOLO el sync final; locks en `outputs/` bloquean la ejecuciГіn entera (gmail.py no puede ni hacer backup).
  вҶ’ REGLA: si aparece lock con mtime >1h sin proceso Excel activo вҶ’ es zombi tras crash, se puede borrar con `rm`.
- **Comando manual requiere `PYTHONPATH=.`** вҖ” `gmail/gmail.py` importa `nucleo.*` y `core.*`. Sin `PYTHONPATH=.` desde la raГӯz del repo, falla con `ModuleNotFoundError: No module named 'nucleo'`. La regla ya estaba documentada para `.bat`/`gmail_auto.bat`, pero el comando manual ad-hoc tambiГ©n la requiere. `docs/FLUJO_MANUAL_GMAIL.md` se corrigiГі 05/05 noche-4 para reflejarlo.
  вҶ’ REGLA: cualquier doc que muestre el comando de gmail.py debe incluir `PYTHONPATH=.` y la ruta `gmail/gmail.py` (no `gmail.py`). Verificar con un test real, no copiar/pegar de memoria.

### AutomatizaciГіn vs ejecuciГіn manual
- **Cron silencioso para flujos que requieren validaciГіn humana** вҶ’ problemas que se acumulan dГӯas/semanas sin que nadie los vea. Caso real (04/05/2026): la factura JALEO 27/04 quedГі zombi 7 dГӯas hasta que se notГі al revisar el log de una ejecuciГіn manual. El extractor faltaba en MAESTRO `ARCHIVO_EXTRACTOR=jaleo.py` desde hacГӯa meses; un cron nocturno lo habrГӯa seguido fallando indefinidamente.
  вҶ’ REGLA: Para flujos donde el log requiere lectura humana caso por caso (zombis, extractores nuevos, formatos PDF cambiantes, fallos OAuth), preferir ejecuciГіn manual con observaciГіn en directo al cron silencioso. DecisiГіn 05/05/2026: `gmail.py` pasa a manual desde PC con Claude Code; `script_barea.py` (ventas, lunes) y backups siguen automГЎticos porque su flujo es mecГЎnico.
  вҶ’ REGLA: Si se reactiva un cron de un flujo "humano", plantar tambiГ©n alerta automГЎtica (email/Slack) cuando aparezca `[DRIVE FALLO]`, `Extractor no encontrado` o `ALERTA ROJA` en el log. Sin alerta, el cron es peor que la ejecuciГіn manual.
- **Premisas no verificadas sobre infraestructura** вҶ’ afirmaciones que se propagan entre sesiones. Caso real (05/05/2026): registrГ© en cierre 04/05 y 05/05 maГұana que "el cron del viernes procesarГЎ..." sin haber consultado `crontab -l` ni una vez. AuditorГӯa 05/05 tarde revelГі que el cron NO existГӯa вҖ” solo habГӯa un header huГ©rfano `# Gmail facturas - viernes 03:00` que nunca se llegГі a activar. La premisa contaminГі dos sesiones.

### Python imports вҖ” `ModuleNotFoundError` vs `ImportError`
- **`from package import submodule` con submГіdulo ausente NO lanza `ModuleNotFoundError`** вҶ’ lanza `ImportError: cannot import name 'X' from 'Y'`. Solo `import X.Y` lanza `ModuleNotFoundError`. El segundo hereda del primero, **NO al revГ©s**. `except ModuleNotFoundError` deja escapar el `from..import` caso.
  вҶ’ REGLA: en cГіdigo que tolera mГіdulos opcionales (loaders, plugins, fallbacks), SIEMPRE `except ImportError`, nunca `except ModuleNotFoundError` aislado. Caso real (24/04/2026): `config/loader.py:_from_legacy()` capturaba solo `ModuleNotFoundError`; Streamlit Cloud lanzaba `ImportError` al hacer `from config import datos_sensibles`. Fix: cambiar a `except ImportError` (commit `5530c10`).

### Streamlit Cloud вҖ” deploy y requirements
- **`requirements.txt` canГіnico en Cloud es el JUNTO AL main file**, no el de la raГӯz вҶ’ Streamlit Cloud resuelve `requirements.txt` adyacente a `streamlit_app/app.py`. Libs aГұadidas SOLO al de la raГӯz NO llegan a Cloud.
  вҶ’ REGLA: al aГұadir libs que Streamlit use, editar `streamlit_app/requirements.txt` (o ambos si se quiere mantener paridad con dev local/VPS). Ver SPEC В§13.10 para deuda de consolidaciГіn.
- **Warning "More than one requirements file detected" en logs de Cloud NO es inofensivo** вҶ’ indica que Cloud estГЎ eligiendo uno distinto al esperado. Leer el log con cuidado.
- **Logging visible en UI ahorra iteraciones de debug** вҶ’ patrГіn `try/except ImportError` con `st.error(...)` + `st.expander("Traceback")` + `st.code(traceback.format_exc())`. Caso real: el 4Вә bug de la cadena `/documentos` v2 se resolviГі en 1 minuto gracias a este logging (el traceback literal lo enviГі el usuario).
  вҶ’ REGLA: pГЎginas Streamlit con imports de dependencias externas (Drive, APIs, etc.) deben envolver con este patrГіn al menos en la primera release.
- **CachГ©s bytecode `.pyc` agresivos en Cloud** вҶ’ a veces un redeploy normal no los limpia. Si un fix no aplica tras push, probar commit vacГӯo o reboot manual. Validar visualmente antes de asumir que funciona.

### Python path en apps multipГЎgina Streamlit
- **`sys.path` centralizado en main file** вҶ’ cada pГЎgina hacГӯa su propio `sys.path.insert(0, ROOT)`, frГЎgil ante pГЎginas nuevas. Centralizar en `streamlit_app/app.py` garantiza que cualquier `pages/<new>.py` hereda el path correcto.
  вҶ’ REGLA: si las pГЎginas importan mГіdulos de la raГӯz del repo (nucleo, config, gmail, etc.), poner el `sys.path.insert(0, ROOT)` en el main file una sola vez.

### Bug nombrado de archivo en facturas multi-albarГЎn
- **gmail.py guarda el PDF con la fecha del PRIMER ALBARГҒN, no la fecha de factura** вҶ’ en facturas que agrupan varios albaranes (tГӯpico ForPlan/MIGUEZ CAL), el nombre de archivo refleja la fecha del primer albarГЎn y desincroniza con la fecha real de factura. Caso real (28/04/2026): factura 31/12/25 archivada como `ATRASADA 4T25 1205 MIGUEZ CAL SL TF.pdf` (12/05 = primer albarГЎn), no como `1231`. DespuГ©s alguien renombra el PDF en disco al nombre correcto (1231) pero la columna ARCHIVO del Excel sigue con 1205 вҶ’ buscar el PDF por nombre exacto falla.
  вҶ’ REGLA: Al investigar incidencias en proveedores con facturas multi-albarГЎn, comparar `ARCHIVO` del Excel con el nombre real del PDF en disco. Si difieren en la fecha, sospechar bug de nombrado en gmail.py.
  вҶ’ REGLA: La FECHA_FACTURA extraГӯda del PDF tambiГ©n puede ser la del primer albarГЎn. El extractor de MIGUEZ tiene este bug (28/04/2026, anotado en backlog). Verificar manualmente facturas multi-albarГЎn hasta arreglar el extractor.

### Bug FORMA_PAGO en flujo gmail.py
- **gmail.py prefiere MAESTRO sobre PDF para FORMA_PAGO cuando el extractor no la extrae** вҶ’ si el extractor no expone `extraer_forma_pago()` o equivalente, el flujo pone en el Excel lo que diga MAESTRO_PROVEEDORES, ignorando lo que estГ© en el PDF. Caso real: DEBORA GARCIA TOLEDANO factura 14/04/26 вҖ” PDF dice "Efectivo", MAESTRO tiene `TJ` вҶ’ escribiГі `TJ` en Excel.

  вҶ’ SOLUCIГ“N PENDIENTE: o (a) que el extractor extraiga FORMA_PAGO del PDF; o (b) que el flujo prefiera SIEMPRE el PDF cuando estГ© presente.

### PatrГіn dual A/B de extractores
- **Dos firmas conviviendo**: la mayorГӯa de extractores definen solo hooks individuales (`extraer_lineas/total/fecha/referencia`) que reciben TEXTO. Solo unos pocos (CERES y similares) definen `extraer(pdf_path)` que devuelve dict con todo. gmail.py:2376-2453 maneja ambos: si la instancia tiene mГ©todo `extraer`, lo invoca; si no, abre el PDF con pdfplumber+OCR y llama a los hooks.
  вҶ’ REGLA: Cualquier herramienta que necesite invocar extractores fuera de gmail.py (p.ej. `scripts/resucitar_zombis.py`) debe replicar este patrГіn dual para mantener paridad. La extracciГіn de texto SГҚ estГЎ expuesta como utilidad reutilizable: `nucleo.pdf.extraer_texto_pdf(ruta, metodo='pdfplumber', fallback=True)`. La selecciГіn de patrГіn A vs B sigue inline en gmail.py вҖ” duplicar con comentario apuntando a la lГӯnea origen.

### Identificadores persistentes vs Гәnicos en `extraer_referencia`
- **Trampa**: Un campo en la factura puede llamarse `Invoice number` o `Customer ID` o `Reference` y ser **persistente del cliente/billing**, no Гәnico por factura. Si un extractor lo captura como REF, el anti-dup CIF+REF (gmail.py:2247-2255) descartarГЎ silenciosamente todas las facturas siguientes del mismo proveedor. Caso real (29/04/2026): `Parseo/extractores/anthropic.py:75-80` capturaba `Invoice number EXB4HCQN` (persistente) en lugar del `Receipt #2880-2984-9190` Гәnico вҶ’ factura Anthropic 20/04 descartada como duplicada, situaciГіn que se repetirГӯa cada mes.
  вҶ’ REGLA: Al crear/revisar un extractor, comprobar con AL MENOS DOS facturas distintas del mismo proveedor que `extraer_referencia()` devuelva valores DIFERENTES. Si devuelve el mismo string para ambas, el patrГіn captura un identificador persistente вҖ” corregir.
  вҶ’ REGLA: Para proveedores con CIF vacГӯo (extranjeros tipo ANTHROPIC), el riesgo es mayor porque `factura_existe` cae al fallback `nombre|REF`, y el `nombre` tambiГ©n es constante вҶ’ la unicidad depende ENTERAMENTE del REF. Doble cuidado.
  вҶ’ DETECCIГ“N: revisar periГіdicamente `datos/emails_procesados.json` por proveedores con N entradas en `emails` pero pocas/una entrada en `facturas` (clave repetida). Es seГұal de REF colisionando.

### Anclar regex de fechas al contexto correcto
- **Cuando un PDF tiene mГәltiples fechas en distintas secciones** (cabecera de factura + tabla de albaranes/lГӯneas/vencimientos), el regex de extracciГіn debe anclarse al contexto de la fecha de cabecera, no capturar la primera que encuentre. Caso de referencia: facturas MIGUEZ CAL multi-albarГЎn contienen `31/12/25 A 4724 ...` en cabecera y `ALBARГҒN A-3184 FECHA 05/12/2025` en tabla вҖ” un regex genГ©rico capturarГӯa la del primer albarГЎn como fecha de factura.
  вҶ’ REGLA: patterns Гәtiles para anclar вҖ” precedido de `Factura`, `Fecha factura`, `F.EmisiГіn`; o restringido a las primeras N lГӯneas; o anclado a un patrГіn Гәnico de la cabecera (ej. `^DD/MM/YY\s+A\s+\d+` para identificar la lГӯnea inmediatamente debajo de "FECHA FACTURA CLIENTE N.I.F TELГүFONO PГҒG.").
  вҶ’ DETECCIГ“N tardГӯa (red de seguridad observacional, gmail.py:_check_fecha_vs_email): si `factura.fecha < internalDate вҲ’ 7 dГӯas` en facturas recientes, el log emite WARNING con la sospecha de multi-albarГЎn. NO bloquea; sГіlo invita a revisar el nombre del archivo manualmente.

### Bug irreproducible вҖ” no tocar

Documentar el caso histГіrico, aГұadir red de seguridad observacional (logs, alertas), y esperar a que el bug reaparezca para diagnosticarlo con datos. Modificar regex "por si acaso" cuando los tests pasan solo introduce riesgo nuevo sin eliminar el viejo.

### Bytes NUL en pdfplumber
- **pdfplumber a veces inserta `\x00` (byte NUL, ord=0) como separador en el texto extraГӯdo**. Caso real (29/04/2026): texto de `Invoice number EXB4HCQN 0006` en PDFs Anthropic вҖ” el "espacio" entre `EXB4HCQN` y `0006` resultГі ser `\x00`, no un espacio real. Un regex con `\s*` falla porque `\s` cubre `[\t\n\r\f\v ]` pero NO el byte NUL.
  вҶ’ REGLA: para campos crГӯticos extraГӯdos del PDF con pdfplumber, los regex deben usar `[\s\x00]*` cuando se permitan separadores. Alternativa: pre-limpiar el texto con `texto.replace('\x00', ' ')` antes de aplicar regex.
  вҶ’ DETECCIГ“N: si un regex que parece correcto cae al fallback inesperadamente, inspeccionar bytes con `for i, c in enumerate(texto[idx:idx+30]): print(i, repr(c), ord(c))` para ver el separador real.
  вҶ’ ГҒMBITO: cualquier extractor que reciba texto de pdfplumber. Casos especialmente sospechosos: PDFs generados por servicios USA (Anthropic, etc.) donde la tipografГӯa y el embedding de texto pueden diferir del estГЎndar europeo.

### Rescate manual de facturas duplicadas zombis
- **Cuando una factura se descarta como duplicado CIF+REF por bug del extractor**, el flujo correcto NO es reprocesar vГӯa gmail.py (eso re-etiquetarГӯa email + intentarГӯa duplicar todo). El flujo correcto es:
  1. Parchear el extractor.
  2. Smoke test contra PDFs histГіricos del proveedor para confirmar que extrae REFs Гәnicas.
  3. Re-parsear el PDF original con el extractor parcheado.
  4. Resucitar manualmente: subir PDF a Dropbox + aГұadir 1 fila a cada Excel canГіnico (PAGOS Drive, Provisional Drive, Provisional Dropbox).
  5. Limpiar el control DB del VPS (`emails_procesados.json`):
     - Borrar la clave zombi de `facturas`.
     - Insertar la clave nueva Гәnica.
     - Reemplazar la entrada de `emails[email_id]` (de `motivo: "Duplicado..."` al formato canГіnico de 5 campos: `message_id, fecha_proceso, proveedor, archivo, dropbox`).
     - NO tocar `ultima_ejecucion`.
  6. Deploy del extractor parcheado al VPS (scp + verificar md5 match).
  вҶ’ REGLA: NO re-etiquetar el email Gmail con label `FACTURAS` para que el cron lo reprocese вҖ” eso intentarГӯa duplicar el PDF en Dropbox (saltarГӯa por hash dup pero contaminarГӯa logs) y crearГӯa una segunda fila Excel con la nueva REF (real) MIENTRAS la fila original errГіnea seguirГӯa sin escribirse. El email permanece en `FACTURAS_PROCESADAS`.
  вҶ’ Caso real (29/04/2026): factura Anthropic 20/04 (REF correcta `EXB4HCQN-0007`, total 90вӮ¬) se rescatГі con script one-shot `scripts/_rescate_anthropic_20260420.py`. Backups del control DB pre/post conservados en `outputs/backups/` como evidencia.

### DiagnГіstico de facturas "perdidas" вҖ” alias coloquial vs canГіnico
- **Cuando el usuario dice "la factura de X no se procesГі"**, antes de buscar bug confirmar que X es exactamente el `nombre` del MAESTRO o uno de sus `alias`. Caso real (29/04): el usuario enumerГі "Pili Blanco" y "Welldone" como no procesadas вҖ” ambas SГҚ estaban procesadas, pero bajo el nombre canГіnico `BLANCO GUTIERREZ PILAR` (alias incluye `PILAR BLANCO`) y `DEL RIO LAMEYER RODOLFO` (alias incluye `WELLDONE`).     1. Email existe en Gmail (consulta API por `from:` o `subject:` libre).

  вҶ’ REGLA: el sistema procesa EL CONTENIDO de un email, no su fecha. Un email del 21/04 puede contener una factura del 13/11/2025 вҖ” el `archivo` reflejarГЎ la fecha real de la factura, no la del email.

### Pandas y tipos de datos
- **`df["col"].dtype == object` es frГЎgil entre pandas 2.x y 3.x** вҶ’ pandas 3.x reporta columnas de strings como dtype `str`, no `object`, rompiendo ramas condicionales que dependen de esa igualdad.
  вҶ’ REGLA: Usar `pd.api.types.is_numeric_dtype(df["col"])` o `pd.api.types.is_object_dtype(...)` en lugar de comparar `.dtype == object`. Invertir la condiciГіn: rama fГЎcil (es numГ©rico) вҶ’ usar directo; rama costosa (cualquier no-numГ©rico, incluido string) вҶ’ limpiar + `pd.to_numeric(errors="coerce")`.
  вҶ’ REGLA: En limpiezas sobre strings de moneda espaГұola, eliminar tambiГ©n `\u00a0` (non-breaking space) y pasar `regex=False` a `.str.replace()` por claridad y velocidad.
- **VPS con versiones de paquetes divergentes de `requirements.txt`** вҶ’ fallos reproducibles solo en producciГіn (caso WC dashboard 20/04/2026: pandas 3.0.2 en VPS, 2.3.0 en PC).
  вҶ’ REGLA: Mantener `requirements.txt` pinneado y exigir al VPS respetar los pins; instalar SIEMPRE con `pip install -r requirements.txt` tras cualquier upgrade.
  вҶ’ REGLA: Tras detectar drift, alinear con `pip install "paquete==X.Y.Z"` y confirmar `pip show paquete` idГ©ntico en ambos entornos.
