## FIX 3 — Import dropbox_api.py

**Problema real encontrado** (distinto al descrito en el playbook):

No era `dropbox_api.py` el que tenía un import roto. La cadena de fallos fue:

1. `from gmail.dropbox_api import DropboxAPIClient` ⇒ dispara `gmail/__init__.py`
2. `gmail/__init__.py` ⇒ `from .auth import GmailConnection, test_conexion`
3. `gmail/auth.py:14` ⇒ `from gmail_config import (...)` — sin prefijo de paquete, falla al importar desde la raíz
4. (tras fix) `gmail/auth.py:21` ⇒ `from config_local import GMAIL_APP_PASSWORD, verificar_credenciales` — `config_local.py` no existe en VPS (contiene credenciales IMAP y por seguridad solo está en PC)

**Cambios aplicados (todos con backup `.bak.20260418`):**

1. **Patrón try/except** en 5 archivos para `from gmail_config import ...`:
   - [gmail/auth.py:14](../gmail/auth.py#L14)
   - [gmail/descargar.py:13](../gmail/descargar.py#L13)
   - [gmail/guardar.py:17](../gmail/guardar.py#L17)
   - [gmail/renombrar.py:20](../gmail/renombrar.py#L20)
   - [gmail/identificar.py:18](../gmail/identificar.py#L18)

   Estructura:
   ```python
   try:
       from gmail.gmail_config import (...)
   except ImportError:
       from gmail_config import (...)
   ```

2. **[gmail/__init__.py](../gmail/__init__.py)** — tolerancia a la ausencia de `config_local`:
   ```python
   try:
       from .auth import GmailConnection, test_conexion
   except ImportError:
       # auth.py requiere config_local (credenciales IMAP) — no disponible en producción VPS (usa OAuth2)
       GmailConnection = None
       test_conexion = None
   ```

Esto permite que módulos del paquete `gmail` funcionen en VPS sin necesidad de distribuir credenciales IMAP.

**Verificación VPS:**
```
from gmail.dropbox_api import DropboxAPIClient  # OK
```
